# План реализации проверки доступности сервера (Offline/Online статус)

Этот план основан на вашем черновике `plan.md` с небольшими техническими дополнениями, чтобы учесть специфику Kotlin Multiplatform (в частности, настройку HTTP-клиента под каждую платформу и доступы к локальной сети).

Мы будем двигаться строго по этапам. Как только вы проверяете и коммитите текущий этап, мы переходим к следующему.

## [ВЫПОЛНЕНО] Этап 1: Подготовка сервера (Health Check)
**Задача:** Научить сервер отвечать на простейший запрос, чтобы мобильные клиенты могли проверять его статус.
- В файл `server/src/main/kotlin/.../Application.kt` добавлен маршрут `GET /api/ping`, который возвращает строку `"pong"` и статус 200 OK.

## [ВЫПОЛНЕНО] Этап 2: Настройка Ktor Client и разрешений для HTTP
**Задача:** Добавить нужные библиотеки для работы с сетью и разрешить локальные HTTP-запросы (без HTTPS) для разработки.
1. Добавление зависимостей в `build.gradle.kts` (в модуль `composeApp` или `shared`):
   - `ktor-client-core` (общий)
   - `ktor-client-okhttp` (Android)
   - `ktor-client-darwin` (iOS)
2. Настройка `AndroidManifest.xml` (Android): `android:usesCleartextTraffic="true"`, чтобы эмулятор мог стучаться по `http://10.0.2.2`.
3. Настройка `Info.plist` (iOS): `NSAppTransportSecurity` -> `NSAllowsArbitraryLoads`, чтобы симулятор мог работать с `http://127.0.0.1` или вашим локальным IP.

## [ВЫПОЛНЕНО] Этап 3: Создание класса NetworkMonitor
**Задача:** Написать логику мониторинга (пинг сервера) в `commonMain`.
1. Реализовать класс `NetworkMonitor`, использующий корутины (`StateFlow` и `HttpClient`).
2. Добавить обработку исключений при отсутствии интернета или выключенном сервере.
3. Организовать периодический пинг (например, раз в 5-10 секунд) или просто однократный при старте/возврате в приложение.

## [ВЫПОЛНЕНО] Этап 4: Интеграция в Compose Multiplatform (UI)
**Задача:** Вывести плашку со статусом сети в приложении.
1. Обновить `App.kt`.
2. Подключить экземпляр `NetworkMonitor`.
3. Сверстать `ServerStatusIndicator`, отображающий цветную плашку (Оффлайн — красный, Онлайн — зеленый, Подключение — серый).
4. Убедиться, что остальной UI приложения корректно отображается независимо от статуса.

## [ВЫПОЛНЕНО] Этап 5: Финальное тестирование
**Задача:** Проверить работу системы в разных сценариях.
1. Запуск с включенным сервером.
2. Отключение сервера "на лету" (если реализуем периодический пинг).
3. Запуск при выключенном сервере (проверка локального режима).
